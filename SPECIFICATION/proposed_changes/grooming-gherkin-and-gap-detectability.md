## Proposal: Make every grooming-realization behavior gap-detectable and testable

### Target specification files

- SPECIFICATION/contracts.md
- SPECIFICATION/scenarios.md

### Summary

The v003 grooming realization (contracts.md §"Grooming and slice-size calibration" + Scenario 7) was written mostly as prose, so the mechanical gap-detector — which keys ONLY on uppercase MUST / SHOULD clause lines — could see almost none of its fundamental behaviors. This proposal makes every fundamental behavior of the realization carry BOTH (a) a normative MUST / SHOULD clause line in contracts.md §"Grooming and slice-size calibration" (so it is gap-detectable) AND (b) a Gherkin scenario in scenarios.md (so it is testable). Prose stays only as augmentation around the clauses and scenarios, never load-bearing alone. Behaviors already carrying MUST clause lines (Dispatcher: refuse human-gated; Dispatcher: non-convergence → needs-regroom + surface; Dispatcher: calibration telemetry; Dispatcher composes next's ranking) gain Gherkin scenarios; behaviors that were prose-only (intake Definition-of-Ready; the groom regroom front-end; the needs-regroom state + transitions; the calibration analysis pass; the single Fabro DOT non-convergence tweak) gain both an explicit clause line and a Gherkin scenario. The two named open realization choices (needs-regroom label-vs-status; groom own-skill vs capture-work-item mode) stay documented-open and unresolved; every scenario is written against observable behavior, not internal representation.

### Motivation

The mechanical gap-detector (detect-impl-gaps) enumerates rules by scanning, line-by-line outside code fences, for the word-boundary keywords MUST / MUST NOT / SHOULD / SHOULD NOT and deriving one gap-id per such line. The v003 realization expressed most of its behaviors as lowercase prose, so the detector cannot see them and capture-impl-gaps cannot file them — the realization is under-enforced. Adding an explicit normative clause line per fundamental behavior makes each behavior a detectable gap that the gap-tracking loop can hold the impl accountable to; pairing each with a Gherkin scenario makes the same behavior bind to an executable acceptance test via the heading-coverage map. This closes the gap between 'the realization is specified' and 'the realization is enforced and testable' without resolving any of the deliberately-open representation choices.

### Proposed Changes

Edit `SPECIFICATION/contracts.md` §"Grooming and slice-size calibration" and `SPECIFICATION/scenarios.md` as follows. All new contract clause lines are PLAIN PROSE LINES (outside any code fence) so the line-oriented gap-detector sees each as exactly one rule; all new scenarios are Gherkin blocks fenced in ```gherkin (so they are testable but are NOT themselves counted as gap rules). Write every scenario against OBSERVABLE behavior (e.g. "the item is surfaced to the maintainer and is not auto-dispatched"), never against internal representation, so neither open choice is pinned.

=== scenarios.md — new Gherkin scenarios ===

- `## Scenario 8 — Intake Definition-of-Ready triage` (behavior 1): a Feature with three Scenario blocks — (a) Given a freshly-described single-acceptance item with one coherent "done", autonomously verifiable, autonomy-tiered, dependency-linked, repo-targeted, above-floor; When it is filed via a capture front-end running the intake Definition-of-Ready checklist; Then it is tagged `ready` and is eligible for autonomous dispatch.
