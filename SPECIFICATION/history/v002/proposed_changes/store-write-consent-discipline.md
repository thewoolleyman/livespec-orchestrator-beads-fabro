## Proposal: Store-write consent discipline

### Target specification files

- SPECIFICATION/contracts.md
- SPECIFICATION/constraints.md

### Summary

Codify the substrate-agnostic store-write consent discipline: state-changing writes to the orchestrator's persistent work-items / memos store performed on the user's behalf MUST be per-operation user-consented unless the user explicitly waived consent for the named operation class. A new contracts.md section names the six heavyweight SKILL.md front-ends as the governed surface, defines the three recognized consent forms, defines the explicit operation-class waiver mechanism, and carves out the Dispatcher's machine-path dispositions (close-on-confirmed-merge of already-filed items with PR + merge-sha audit evidence) as the sole exemption. constraints.md gains matching binary rules under its Skill orchestration constraints section.

### Motivation

Work-item livespec-impl-beads-nip: the per-operation consent principle is currently enforced only by convention inside individual SKILL.md files; it is not stated anywhere in this repo's SPECIFICATION, so nothing prevents a future front-end (or a future edit to an existing one) from auto-writing to the store without consent. Under livespec contracts.md section 'Interactive dialogue ownership (orchestrator-side)' the consent dialogue is orchestrator-owned, and this repo's in-repo SKILL.md front-ends (capture-impl-gaps, capture-memo, capture-spec-drift, capture-work-item, implement, process-memos) are exactly those front-ends. The Dispatcher (dispatcher.py dispatch/loop) writes to the work-items store only as machine-path dispositions of already-filed items, which the principle exempts by design (it acts on items a human or a consented front-end already filed; --no-close-on-merge disables even that) - the codification must carve out this machine path explicitly so the discipline and the Dispatcher do not contradict.

### Proposed Changes

Add a new H2 section `## Store-write consent discipline` to `SPECIFICATION/contracts.md`, placed between `## The ten-skill surface` and `## Beads connection model`, stating: (1) the substrate-agnostic principle - a state-changing write to the orchestrator's persistent work-items / memos store performed on the user's behalf MUST be per-operation user-consented, unless the user has explicitly waived consent for the named operation class; (2) per livespec contracts.md section 'Interactive dialogue ownership (orchestrator-side)', the consent dialogue is orchestrator-owned, and this plugin's six heavyweight SKILL.md front-ends (`capture-impl-gaps`, `capture-memo`, `capture-spec-drift`, `capture-work-item`, `implement`, `process-memos`) are exactly those front-ends - the governed surface; (3) the three recognized consent forms - explicit confirmation (the skill presents the assembled record or pending disposition and the user confirms), consent-by-authorship (the user-supplied free text fully determines the single resulting write and the skill performs no other store write, e.g. capture-memo's deposit - no second confirmation prompt is required), and up-front operation decision (an explicit user decision at the start of the operation that names the write it will perform, e.g. implement's resolution-path decision); consent MUST be obtained before the store write executes; (4) the operation-class waiver mechanism - the user MAY explicitly waive per-operation consent for a named operation class (e.g. 'auto-dispose all pending memos as discard'); a waiver MUST be explicit, MUST name the operation class it covers, and is scoped to the current invocation - it MUST NOT be a default, MUST NOT be inferred, and MUST NOT persist across sessions; absent a waiver, per-operation consent is required; (5) the machine-path exemption - the Dispatcher (`dispatcher.py` `dispatch` / `loop`) writes to the work-items store ONLY as machine-path dispositions of already-filed items (close-on-confirmed-merge carrying PR-number and merge-sha audit evidence in the AuditRecord); these writes are EXEMPT from the per-operation consent discipline by design, because the Dispatcher acts on items a human or a consented front-end already filed, and `--no-close-on-merge` disables even that path; the exemption covers ONLY dispositions of already-filed items - the Dispatcher MUST NOT create net-new work-items or memos on its own initiative; the Dispatcher's module docstring documents this boundary; (6) out-of-scope surfaces - the four thin-transport skills are query-only by contract and never write to the store, and `capture-spec-drift` writes nothing to this plugin's store (its output is a per-finding user-consented `/livespec:propose-change` handoff). Additionally, append two binary rules to `SPECIFICATION/constraints.md` section `## Skill orchestration constraints`: heavyweight skills that write to the work-items / memos store MUST obtain per-operation user consent before each store write in one of the recognized consent forms, unless the user explicitly waived consent for the named operation class (per contracts.md section 'Store-write consent discipline'), with waivers explicit, class-named, and invocation-scoped; and the Dispatcher's machine-path dispositions of already-filed items are the SOLE exemption, with `--no-close-on-merge` disabling even that path, and the Dispatcher MUST NOT create net-new work-items or memos.
